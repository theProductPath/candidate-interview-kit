## Overview

Build the Team Kit edition of the Candidate Interview Kit. This extends the Individual Kit to support multiple interviewers evaluating the same candidates for the same role, coordinated by a hiring manager.

**The Individual Kit is the reference implementation.** The Team Kit follows the same conventions — markdown files as source of truth, Node.js scripts for automation, self-contained HTML comparison tool — with extensions for multi-interviewer workflows.

## What's New in Team Kit (vs. Individual)

1. **Multiple interviewers** — Each interviewer gets their own brief and assessment per candidate
2. **Hiring team file** — `_team.md` defines the roster with per-interviewer focus areas
3. **Progressive prep** — When generating a brief for one interviewer, the skill incorporates findings from any *existing* assessments by other interviewers for that candidate
4. **Per-interviewer attribution** — Every qualitative score and piece of evidence is tagged to the interviewer who produced it
5. **Interviewer toggles in comparison tool** — The hiring manager can turn each interviewer's qualitative input on/off and watch scores shift
6. **Separate comparison template** — `comparison-team-template.html` with team-specific UI (interviewer toggles, per-interviewer evidence, two-level filtering)

## Folder Structure

Each candidate has a subfolder per interviewer under their candidate folder. The resume lives at the candidate level (shared, not duplicated). Briefs and assessments live inside the interviewer's subfolder.

```
{Company}-{Role}/
├── _team.md                              ← Hiring team roster + per-interviewer focus areas
├── job-description.md                    ← JD text (source of truth, static, shared)
├── candidates/
│   └── {candidate-name}/
│       ├── resume.pdf                    ← Shared resume (one copy)
│       └── {interviewer-slug}/
│           ├── brief.md                  ← AI-generated prep brief for THIS interviewer
│           └── assessment.md             ← AI-structured post-interview assessment
└── comparison.html                       ← Team comparison Burner App (generated by skill)
```

```

├── candidates/
│   ├── maya-torres/

│   │   │   ├── brief.md
│   │   │   └── assessment.md
│   │   ├── sarah-chen/
│   │   │   ├── brief.md
│   │   │   └── assessment.md
│   │   └── marcus-lee/
│   │       ├── brief.md
│   │       └── assessment.md
│   └── james-okafor/
```

**Key conventions:**
- Interviewer subfolder names are slugified from the interviewer's full name (same `slugifyCandidateName` logic — reuse it, maybe rename to `slugify`)
- Resume lives at `candidates/{candidate}/resume.pdf` — never duplicated into interviewer folders
- Briefs and assessments use the same filenames as Individual Kit (`brief.md`, `assessment.md`) — the interviewer context comes from the parent folder name
- The `_team.md` file must exist before the skill can generate briefs or assessments

## `_team.md` — Hiring Team Definition

A single human-readable markdown file that defines the hiring team and per-interviewer focus areas. Written by the hiring manager (or setup person) at kit initialization time.

### Format

```markdown
# Hiring Team
**Role:** {Role Title}
**Company:** {Company}
**Hiring Manager:** {Name}

## {Interviewer Full Name} — {Title}
**Role on team:** Hiring Manager
**Focus areas:**
- {Focus area 1 — what this interviewer should probe}
- {Focus area 2}
- {Focus area 3}

## {Interviewer Full Name} — {Title}
**Focus areas:**
- {Focus area 1}
- {Focus area 2}

## {Interviewer Full Name} — {Title}
**Focus areas:**
- {Focus area 1}
- {Focus area 2}
```

### Parsing rules

- Each `## ` heading defines an interviewer
- The heading format is `## {Name} — {Title}` (em dash separator)
- `**Role on team:** Hiring Manager` is optional — identifies the HM. Only one interviewer should have this. If none is tagged, default to the name in the top-level `**Hiring Manager:**` field
- `**Focus areas:**` followed by a bullet list defines what the skill should emphasize in that interviewer's prep briefs
- The hiring manager IS a full interviewer — they get briefs and assessments like everyone else
- No "Order" field. Interview sequence is not prescribed.

### Example

```markdown
# Hiring Team
**Role:** Product Analyst — Platform Modernization
**Company:** AirCorp
**Hiring Manager:** Jordan Blake

## Jordan Blake — Engineering Director
**Role on team:** Hiring Manager
**Focus areas:**
- Migration experience depth — have they actually hands-on led a platform or data migration?
- Stakeholder communication under pressure — how they handle disagreement between technical and business stakeholders
- Ambiguity tolerance — do they adapt when requirements shift, or do they stall?
- Documentation discipline — are they genuinely good at this, not just claiming to be?

## Sarah Chen — Sr. Product Manager
**Focus areas:**
- Product thinking — do they think about user outcomes or just process execution?
- Cross-industry adaptability — can they transfer skills from one domain to another?
- Quantitative analysis depth — can they work with data, or are all examples qualitative?
- Requirements gathering maturity — is their process structured and repeatable?

## Marcus Lee — Tech Lead
**Focus areas:**
- Technical depth — can they discuss system architecture beyond surface-level tool names?
- System decomposition thinking — do they question why components exist, or just document them?
- Architecture awareness — do they understand how systems connect and where integration points matter?
- Tool proficiency beyond resume claims — push past the buzzword list
```

## SKILL.md — Team Kit Skill

Create `team-kit/SKILL.md`. This is the AI skill file for the team edition. It follows the same structure and conventions as `individual-kit/SKILL.md` with the following differences:

### Identity

```
You are an interview intelligence assistant for a hiring team working through a shared hiring loop. You help each interviewer prepare for their interviews, structure their post-interview assessments, and produce a team comparison tool that lets the hiring manager consolidate input from every interviewer.
```

### Working folder awareness

The skill must understand the team folder structure:
- `_team.md` is required (equivalent to how `job-description.md` is required in individual)
- Each candidate has interviewer subfolders under their candidate directory
- Resume is shared at the candidate level; briefs and assessments are per-interviewer

### How to determine what to do

Same pattern as individual, with team-specific commands:

| Command | Action |
|---|---|
| "Initialize" or "Setup" | Scaffold team kit |
| "Add candidate: [name]" | Create candidate folder (no interviewer subfolders yet — those are created on first brief) |
| "Add interviewer: [name] — [title]" | Add an interviewer section to `_team.md` with prompted focus areas |
| "Generate brief for [candidate] for [interviewer]" | Step 1 — Team Brief |
| "Post-interview: [candidate] for [interviewer]" | Step 2 — Team Assessment |
| "Update comparison" | Step 3 — Team Comparison |

Auto-detection (scan folder state):

| What you see | What to do |
|---|---|
| No `_team.md` | Offer setup |
| `_team.md` exists, no candidates | Ask to add candidates |
| Candidate folder exists with resume, interviewer has no `brief.md` | Offer to generate brief for that interviewer |
| Interviewer has `brief.md` but no `assessment.md` | Ask for interview notes or offer to generate assessment |
| Multiple interviewers have assessments | Offer to refresh comparison |

### Step 1 — Generate Prep Brief (Team)

**Read:**
- `job-description.md`
- `_team.md` — specifically the section for the target interviewer (their focus areas)
- `candidates/{candidate}/resume.pdf`
- **All existing assessments** from OTHER interviewers for this same candidate (`candidates/{candidate}/{other-interviewer}/assessment.md`) — this is the progressive prep mechanism

**Produce:** `candidates/{candidate}/{interviewer}/brief.md`

**Brief structure — same as individual with these additions:**

```markdown
# Interview Prep Brief — {Candidate Name}
*Role: {Role} | {Company} | Interviewer: {Interviewer Name}, {Title} | Generated: {date}*

---

## Candidate Snapshot
[Same as individual — 3-4 sentence summary from resume]

## JD Competency Assessment
[Same table as individual — resume vs JD scoring]

## Your Focus Areas
Based on the hiring manager's direction for your interview, prioritize:
[Bulleted list drawn directly from this interviewer's focus areas in _team.md, mapped to this candidate's resume]

## Recommended Probe Areas
[2-4 areas where resume is thin or unclear, tailored to this interviewer's focus]

## Suggested Questions
[5-7 targeted questions — tailored to BOTH the candidate's resume AND this interviewer's focus areas]

## Watch For
[1-3 flags specific to this interviewer's focus]

## Findings from Prior Interviews
[ONLY present if other interviewers' assessments exist for this candidate]
[Summarize key findings — confirmed strengths, flagged gaps, open questions]
[Tag each finding with the interviewer who surfaced it]
[Suggest specific follow-up areas based on what earlier interviewers found]
```

**Progressive prep rules:**
- If no other assessments exist yet, omit the "Findings from Prior Interviews" section entirely — don't include an empty section
- If assessments exist, read them and summarize: confirmed strengths, flagged gaps, and areas the current interviewer should follow up on
- The progressive prep enriches the brief — it does NOT replace the interviewer's own focus areas
- Never reveal one interviewer's raw scores to another — share findings and evidence, not numerical scores

### Step 2 — Structure Post-Interview Assessment (Team)

**Read:**
- `job-description.md`
- `_team.md` — the section for this interviewer
- `candidates/{candidate}/{interviewer}/brief.md`
- User's raw interview notes

**Produce:** `candidates/{candidate}/{interviewer}/assessment.md`

**Assessment structure — same as individual with one addition to the header:**

```markdown
# Interview Assessment — {Candidate Name}
*Role: {Role} | {Company} | Interviewer: {Interviewer Name}, {Title} | Date: {date}*
```

The rest of the assessment structure is identical to the individual kit. The interviewer attribution comes from the folder location and the header metadata.

**After writing:** Same preview behavior as individual, then refresh comparison.

### Step 3 — Generate Team Comparison Tool

**Read:**
- `job-description.md`
- `_team.md` (full roster)
- All `brief.md` files (for JD competency scores — these come from resume analysis and should be consistent across interviewers, but use the first available)
- All `assessment.md` files (for per-interviewer qualitative scores)

**Produce:** `comparison.html` from `skill/assets/comparison-team-template.html`

## Team Comparison Tool — `comparison-team-template.html`

This is a separate template from the individual kit's `comparison-template.html`. It builds on the same design system (The Sibling + Document Layer) but adds team-specific features.

### Design reference

**The individual kit's `comparison-template.html` is the design authority.** The team comparison tool must match the same look and feel — same design system (The Sibling + Document Layer), same card layouts, same radar chart style, same typography and color tokens. Do not redesign. Extend the individual template with team-specific features (interviewer toggles, per-interviewer evidence attribution, two-level filtering) while preserving the established visual language.

The team prototype in the SMJ incubator (`~/Google Drive/My Drive/AI-Collab/smj/offerings/Candidate-Interview/interview-team-prototype.html`) is a *functional* reference for how the team-specific interactions work (interviewer toggles, progressive prep tabs, per-interviewer evidence tagging). Use it for interaction patterns and data flow — not for visual design. The individual kit's shipped comparison template is the visual standard.

### UI requirements

**1. Interviewer toggle panel**
- "Resume Analysis" pill — always on, locked, cannot be toggled (JD scores are static)
- One pill per interviewer — toggleable on/off
- Each pill shows: interviewer name, title, and whether they're the hiring manager
- Color-coded to match the interviewer's assigned color
- When toggled, all qualitative scores, rankings, and evidence update in real time

**2. Candidate toggle panel**
- Same as individual — select up to 3 candidates for radar chart
- Show interview status per candidate (how many interviewers have assessed)

**3. Summary cards**
- Per-candidate cards showing: JD score, QL score (averaged across active interviewers), combined score
- QL label dynamically shows count: "QL (2 interviewers)" / "QL (1 interviewer)" / "QL (none)"
- When no interviewers are active, QL shows "—" and combined = JD only

**4. JD Competency Radar Chart**
- Same as individual — pure SVG, no external libraries
- Shows JD competency scores (static, unaffected by interviewer toggles)
- Candidate color polygons overlap at 0.7 opacity

**5. Detailed breakdown — two sections**

*JD Competencies (Resume Analysis):*
- Same as individual — expandable rows with evidence
- Not affected by interviewer toggles

*Qualitative Dimensions (Interviewer Assessments):*
- Score bars show averaged score across active interviewers
- Expandable evidence panel shows evidence **tagged by interviewer** (interviewer name badge, color-coded)
- When interviewers are toggled off, their evidence disappears and scores recalculate
- This is the two-level filtering: (a) which candidates are shown, (b) which interviewers' input is included

**6. Ranking bar**
- Combined score ranking across selected candidates
- Updates when interviewer toggles change (because QL averages shift)
- Shows: rank position, candidate name, combined score, JD score, QL score with interviewer count

**7. Partial-state handling**
- Not all interviewers will have assessed all candidates at any given time
- Candidate status levels: `not-started` (folder only), `pending` (has brief from at least one interviewer, no assessments), `partial` (some interviewers assessed, not all), `assessed` (all interviewers assessed)
- Partial candidates should still appear in the comparison with whatever data exists
- Clearly indicate which interviewers have/haven't assessed each candidate

### Design system

Same as individual kit — The Sibling (dark header) + Document Layer (warm parchment content area). Use the exact same CSS variables, colors, and typography defined in the individual kit's SKILL.md design system section. The team prototype HTML also has these variables — reference either source.

## Scripts

All scripts live in `team-kit/skill/scripts/`. They follow the same patterns as the individual kit scripts — Node.js, no external dependencies, run from the kit root.

### `init-team-kit.js`

Scaffolds a new team kit folder. Similar to `init-kit.js` but for team context.

**Creates:**
- `candidates/` directory
- `SKILL.md` (copied from team-kit source)
- `skill/` directory (copied from team-kit source)
- `_team.md` — template with placeholder content:

```markdown
# Hiring Team
**Role:** [Role Title]
**Company:** [Company Name]
**Hiring Manager:** [Hiring Manager Name]

## [Interviewer Name] — [Title]
**Role on team:** Hiring Manager
**Focus areas:**
- [What should this interviewer focus on?]
- [What specific areas should they probe?]

## [Interviewer Name] — [Title]
**Focus areas:**
- [What should this interviewer focus on?]
- [What specific areas should they probe?]
```

- `START-HERE.md` — team-specific getting started guide

**Does NOT create:** `comparison.html` (created on first refresh)

### `add-candidate.js`

Same as individual — creates `candidates/{slug}/` directory. Does NOT create interviewer subfolders (those are created when briefs are generated).

Reuse the individual kit's script logic. The team version just needs to be aware it's running in a team context (check for `_team.md` existence).

### `add-interviewer.js` *(new)*

**Usage:** `node skill/scripts/add-interviewer.js "Full Name" "Title" [--hiring-manager] [--focus "area1" --focus "area2"]`

**Action:**
- Appends a new interviewer section to `_team.md`
- If `--hiring-manager` flag is set, adds `**Role on team:** Hiring Manager`
- If `--focus` flags provided, adds them as bullet points under `**Focus areas:**`
- If no focus areas provided, adds placeholder bullets
- Does NOT create interviewer subfolders under candidates (those are created on-demand when briefs are generated)

**Validation:**
- Check that `_team.md` exists (error if not)
- Check that an interviewer with the same slugified name doesn't already exist in the file (error if duplicate)

### `kit-data.js` (extended)

This is the most significant extension. The individual kit's `kit-data.js` builds a data model from candidate markdown files. The team version must:

1. **Parse `_team.md`** — extract interviewer roster (name, title, role, focus areas)
2. **Discover per-interviewer files** — for each candidate, look for `{interviewer-slug}/brief.md` and `{interviewer-slug}/assessment.md`
3. **Build per-interviewer assessment records** — reuse `parseBrief()` and `parseAssessment()` from individual kit (the markdown format is the same)
4. **Compute candidate status** — not-started / pending / partial / assessed based on how many interviewers have briefs/assessments
5. **Build team data model** — the JSON structure defined in the comparison tool section above

**Key function: `buildTeamKitData(kitRoot)`**

Returns the full team data object. This replaces `buildKitData()` for team context.

**JD competency scores:** These come from resume analysis and should be the same regardless of which interviewer generated the brief. Use the first available `brief.md` found for each candidate to extract JD scores. If multiple interviewers have briefs with slightly different JD scores (because the AI generated them independently), use the first one found alphabetically by interviewer slug. The JD score represents resume-vs-JD analysis, not interview performance.

**Qualitative scores:** These are per-interviewer. Each interviewer's `assessment.md` has its own qualitative dimension scores. The data model stores them separately per interviewer. Averaging happens client-side in the comparison tool based on which interviewers are toggled on.

### `refresh-comparison.js` (adapted)

Same pattern as individual — reads template, builds data, injects JSON, writes `comparison.html`.

**Changes:**
- Uses `comparison-team-template.html` instead of `comparison-template.html`
- Calls `buildTeamKitData()` instead of `buildKitData()`
- Validates that `_team.md` exists (in addition to `job-description.md`)

### `source-discovery.js` (extended)

Add discovery for `_team.md`:
- Look for `_team.md` in kit root
- Add to the sources object returned by `discoverSources()`
- Add validation in `validateKitPrereqs()` — `_team.md` is required for team kit

### `rename-candidate.js` (adapted)

Same as individual but must also rename interviewer subfolders' content if the candidate folder name changes. The interviewer subfolders move with the candidate folder (they're nested inside it), so a simple `fs.renameSync` on the candidate folder should handle it. Just update title lines in all `brief.md` and `assessment.md` files found in any interviewer subfolder.

### Shared utilities

Both individual and team kits use the same markdown parsing functions (`parseBrief`, `parseAssessment`, `extractTable`, etc.). Consider:
- Keep `kit-data.js` in the team kit as its own file that imports or duplicates the parsing utilities
- OR extract shared parsing into a common module both kits can use

**Recommendation:** For now, duplicate the parsing functions into the team kit's `kit-data.js`. Refactoring into shared modules can happen later — keeping each kit self-contained is more important at this stage than DRY.

## What NOT to Build

- No `_project.json` — `_team.md` is the single source of team configuration
- No interview ordering/sequencing logic — interviews happen in whatever order they happen
- No inter-interviewer score sharing in briefs — share findings and evidence, not numerical scores
- No changes to the Individual Kit — it stays exactly as-is
- No login, authentication, or multi-user file locking — this is a folder-based tool
